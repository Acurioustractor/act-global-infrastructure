### CONFIRMED — The Xero Accounting API CANNOT mark a bank statement line / bank transaction as reconciled — reconciliation (matching a statement line and clicking OK) is only possible in the Xero web UI, not via the API.

CONFIRMED after adversarially trying to refute it. I searched specifically for an official reconcile endpoint and found instead an explicit, dated official refusal to provide one.

DECISIVE PRIMARY SOURCE (read directly, twice, consistent results): Xero's own admin response on the long-standing "Reconcile via the API" feature request. Status = DECLINED. Most recent official Xero staff response: "Kelly (Admin, Xero Developer Centre)", dated 6 May 2026. Verbatim official statement: "we will not be adding the ability to reconcile bank statement lines via the API or to expose unreconciled bank statement data via the public API." Stated rationale: unreconciled bank statement lines are treated as raw banking data under consumer data rights regimes, so Xero will not redistribute them to third-party apps, and it will not recreate the reconciliation workflow outside the product. This is the canonical, authoritative refutation of the existence of any reconcile-via-API capability — it is Xero stating, as policy, that the capability does not and will not exist via the public API.

CORROBORATION 1 (Xero Central, UI-only nature): The "Manually mark a transaction as reconciled" article (central.xero.com/.../Reconcile-an-account-transaction-without-an-imported-bank-statement) explains you can only manually mark a transaction reconciled when you CANNOT import bank statement data, and Xero then creates a NEW statement line to reconcile against. The genuine reconcile action — matching an imported bank statement line and clicking OK in the Reconcile screen — is therefore a UI workflow, not an API operation. (Surfaced via WebSearch summary; the article page itself timed out on direct fetch.)

CORROBORATION 2 (field mechanics, from prior OpenAPI-spec research + search snippets): IsReconciled on BankTransactions is a status/reporting boolean. The API can CREATE spend/receive money and bank transfers, but these land as UNRECONCILED entries awaiting manual match in the UI (FromIsReconciled/ToIsReconciled default false on transfers). For standard private/custom apps, IsReconciled cannot be used to reconcile a real imported bank statement line. Consistent with the ACT memory note "Xero API CANNOT set IsReconciled=true (UI-only)".

NUANCE (does not change the verdict): The claim is precise as stated for normal apps. The only documented exception is narrow and does not contradict the claim's spirit — conversion-related apps can set IsReconciled in scenarios where a matching bank statement line is NOT available (i.e., not the real reconcile-against-an-imported-statement-line action). That exception confirms, rather than undermines, that there is no API path to reconcile an actual imported statement line.

CONFIDENCE: The uservoice official admin response is VERIFIED (read the primary source directly, twice, identical substance). The field-mechanics detail and the Xero Central UI-only article are INFERRED from search summaries plus prior OpenAPI-spec reads — the developer.xero.com BankStatements and BankTransactions doc pages were unreachable (60s timeouts / ECONNREFUSED across 4 attempts; firecrawl out of credits), so I could not quote those two doc pages verbatim. That gap does not weaken the verdict: the single most authoritative artefact — Xero stating as policy it will not add reconciliation via the API — directly supports the claim.

### CONFIRMED — Xero has NO public API for creating or managing Bank Rules — bank rules can only be created in the Xero UI.

CONFIRMED after adversarially trying to refute it (searched for a bank-rules endpoint, checked the changelog/release notes for any recent addition, and grepped the full official OpenAPI spec for any "rule" resource — found nothing that contradicts the claim).

1. OpenAPI spec (strongest, directly verified). I downloaded the complete official Xero Accounting OpenAPI spec (913,267 bytes) from https://raw.githubusercontent.com/XeroAPI/Xero-OpenAPI/master/xero_accounting.yaml and grepped it. There are ZERO occurrences of the string "rule" anywhere in the 913KB spec, and zero "/BankRules" path. The full top-level path list contains only banking resources /BankTransactions and /BankTransfers (plus Accounts, Invoices, Contacts, Payments, Reports, etc.) — no Bank Rules resource of any kind. You cannot list, get, create, update, or delete a bank rule via the Accounting API because no such resource is defined. (Verified — read the spec directly.)

### CONFIRMED — "Therefore any tool that 'auto-reconciles' Xero either (a) only pre-creates/pre-codes transactions so the UI match is one-click, (b) relies on bank rules set up manually, or (c) drives the UI via browser automation — none can truly reconcile head-less via the official API."

## Retro — what worked
- Tracer-bullet discipline: one smallest high-confidence delete (Apple $11.99) proved the full delete→verify→keeper-survives path before any batch ran
- Live single-GET re-verification of IsReconciled on every txn before a write — the mirror and the batch IDs= endpoint both lie on recon status, only BankTransactions/{id} is truth
- Hard abort gates on every Tier-3 write: SPEND in NAB Visa #8815, live unreconciled, keeper bill PAID + HasAttachments + amount-within-surcharge, and keeper's payment itself reconciled — no gate, no write
- Reference-based dedup: only treating 2+ lines as a TRUE duplicate when they share the same non-empty bank Reference; different/no ref = separate charges, flagged not deleted
- Void-bill-keep-spend-money for the ambiguous cluster — 48 phantom AUTHORISED bills cleared (~$106.9K phantom AP) BAS-neutral, card lines now match the real spend-money
- Attempted-vs-actual reconciliation on every write (re-read live, confirm the void/delete stuck AND the keeper survived), with a complete append-only audit trail in reconcile-write-log.md
- API recodes of account/tax/project on spend-money before the human reconcile pass — cleans coding even though the API can't click reconcile
- MATCH-THEN-DEDUPE worklist split that quarantined the 8-line DANGER CLUSTER (matches an unpaid AUTHORISED bill) away from the safe-delete bucket

## Retro — what failed
- Archived-contact void DEAD-END: Xero blocked the void on 'contact archived' and the API rejected both un-archiving and a direct void; only workaround was restore-contact → void → re-archive (and an active duplicate 'The Sand Yard' contact complicated it) — pushed to a manual UI Phase-0 list
- The Supabase mirror is unreliable for go/no-go: it conflates spend-money/bills/statement lines and is stale on status (RNM showed AUTHORISED in mirror but was VOIDED live; Belong recon mirror=false vs live=true)
- Batch BankTransactions?IDs= endpoint does not reliably return IsReconciled — forces a slow per-line single GET (rate-limited ~1100ms) for the one flag that makes a delete safe
- Initial duplicate-GST estimate was wrong (~$1,651) — actual 1B over-claim was $740.69 once measured per-line; over-stated before the tracer corrected it
- Two writes (Kennards $424 delete, Dialpad $56 void) succeeded live but the auto-log append missed — a transient verify error and a 429 on the confirm GET — had to be hand-confirmed and hand-logged

## Retro — key learnings
- The mirror is fine to GENERATE candidates but must NEVER AUTHORISE a write — every Tier-3 delete/void is gated against a fresh live single-GET BankTransactions/{id}; the batch IDs= endpoint and the cached mirror both lie on IsReconciled
- Three distinct shapes needed three distinct actions, and conflating them is the expensive error: (a) phantom spend-money with a PAID keeper bill → DELETE; (b) phantom AUTHORISED bill with a real card spend-money → VOID the bill, keep the spend-money; (c) line matching an unpaid AUTHORISED bill → MATCH (deleting falsely un-pays a real bill and understates spend)
- Same amount + date + vendor is NOT a duplicate — only a shared non-empty bank Reference proves one charge recorded twice; the 3 D-group false matches (Uber, Snowdon, Railway) prove coincidental-amount deletes would have destroyed real, separate charges
- On Dext duplicates the receipt + correct coding sit on the unreconciled copy you'd instinctively delete — blind-deleting the card txn bins the substantiation; keep whichever copy holds the attachment and reconcile to it
- The API can recode account/tax/project and can void/delete, but CANNOT set IsReconciled — the final reconcile click is UI-only, so all of this is 'prepare everything, hand Ben a clean 1:1 reconcile tab', not 'reconcile via API'
- GST-free big-dollar duplicates carry the highest receipt risk and ZERO BAS upside — defer them to a deliberate hygiene pass; only the GST-bearing lines (27 of 38, $740.69) actually move the BAS and are worth touching before lodgement
- Archived contacts can hard-block a void via the API — that path is a known dead-end and belongs on a manual UI list, not in an automated batch
- Auto-log on success can be silently lost on a transient verify/429 error even when the live write succeeded — the live re-read is the source of truth, and the log must be reconciled against live afterward
