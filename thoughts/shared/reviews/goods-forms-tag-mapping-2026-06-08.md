## GHL write architecture

**Single chokepoint module:** `src/lib/ghl/index.ts`. Every contact create/update funnels through one private function `createOrUpdateContact()` (`index.ts:546`), exposed only through the `ghl` object's public methods (`index.ts:1030`). There is no second GHL client; nothing in `src/` POSTs to `services.leadconnectorhq.com` except via this module.

- **Base URL:** `https://services.leadconnectorhq.com` (`index.ts:14`), API version header `2021-07-28` (`index.ts:366`).
- **Auth / location:** `GHL_API_KEY` (Bearer) + `GHL_LOCATION_ID` env vars (`index.ts:12-13`). The location id is NOT hard-coded in source — it comes from env. (Per ACT memory `[[website-forms-ghl-one-account]]`, the live location is the one "A Curious Tractor" account `agzsSZWgovjwgpcoASWG`; the opportunity-model custom-field IDs at `index.ts:410-427` confirm this is the shared ACT location.)
- **Feature flag:** `GHL_ENABLED = GHL_ENABLED==='true' && !!API_KEY && !!LOCATION_ID` (`index.ts:19`). When false, all writes are simulated (logged, no API call).
- **Dedupe:** `findContact()` searches by phone-or-email via `/contacts/search/duplicate` before create vs update (`index.ts:527`, `568-570`). Phone takes precedence over email as the dup key.
- **Tag vocabulary (CURRENT):** a flat `goods-*` / `act-*` / `project-goods` string convention defined in the `TAGS` map (`index.ts:159-186`) plus dynamically-built tags (`goods-src-<tag>`, `goods-segment-<x>`, `goods-tier-<x>`, `goods-timeline-<x>`, `goods-asset-<slug>`, `goods-user-request-<type>`). **None of these use the canonical `project:` / `role:` / `comms:` colon-namespaces.** This is the core drift: Goods uses its own pre-taxonomy `goods-`/`project-goods` scheme, NOT the ACT namespaced contract.
- **Project stamp (CURRENT):** every Goods write injects a `withGoodsProject()` custom field = `projectDesignation` (a SHARED dropdown custom field, env `GHL_FIELD_PROJECT_DESIGNATION`) set to the string `"Goods"` (`index.ts:39-44`). So Goods identity is carried as a *custom-field value* ("Goods"), plus the flat tag `project-goods` on the contact/partnership/feedback/story paths — NOT as a `project:act-gd` tag.
- **Routing model:** "code identifies-and-tags, GHL decides what to send." One `smartRouter` workflow (`GHL_WORKFLOW_SMART_ROUTER`) is fired after most writes and branches in-GHL on the `goods-*` tags; retail orders fire a dedicated `newOrder` workflow instead (`index.ts:58-73`, `1191-1200`). New event = new tag value + new GHL branch, zero code change (documented decision 2026-05-22, `index.ts:46-57`).
- **Custom fields written:** order number / order total / product type / community / message / asset id / projectDesignation (`CUSTOM_FIELDS`, `index.ts:22-34`), all by env-configured field IDs. A second set of **opportunity-model** field IDs (`GOODS_OPP_FIELD_IDS`, `index.ts:410-427`) is READ-only here (money-alignment Layer A) — not written by any form.

### Consent / OCAP posture (CURRENT) — the two flags to carry into §E

1. **No explicit newsletter-consent field is ever captured or written.** `addToNewsletter()` (`index.ts:1432`) grants the send-trigger tag `goods-newsletter` to anyone who submits an email/phone, with NO `newsletter_consent=Yes` field and no consent checkbox upstream (the `NewsletterSignup` component `src/components/newsletter-signup.tsx` and `canberra/follow-form.tsx` collect email/phone/name only). The act of submitting the form is treated as consent. Under the canonical Layer-5 gate this is *implied* consent, not explicit.
2. **No `lane:community` / OCAP gate exists anywhere.** The recipient-claim path (`createRecipientContact`, `index.ts:1505`) and the bed-story path (`/api/bed/[id]/story`) write community-line people (recipients, storytellers) into GHL. Neither grants a `comms:*` send tag today, so there is no *active* drip-enrolment of a community member — but there is also no protective `lane:community` marker to *prevent* a future Smart-Router branch or a hand-added tag from enrolling them. See per-form rulings.
