# Living Ecosystem System - Readiness Synthesis

> [!abstract] Short answer
> The system is close enough to shift from plumbing-first work into real design and integration work. It is not finished, but it is past the point where more backend/admin cleanup is the best use of attention.

## Current Read

### Verified

- The knowledge core is real:
  - canonical wiki structure exists
  - canon registry exists
  - source-packet schema exists
  - source-bridge and provenance templates exist
- The integration seam is real:
  - Empathy Ledger source packets now build and serve on production
  - ACT hub now consumes EL packet data in live builds
  - production hub pages render packet-derived media
- The public shell is real:
  - ACT hub has a coherent public frame
  - wiki, projects, art, method, ecosystem, and flagship detail pages already exist
  - Empathy Ledger is content-synced to the same flagship framing

### Inferred

- The system is roughly:
  - **knowledge/information architecture:** 75-85% ready
  - **integration/runtime seam:** 75-85% ready for ACT internal use
  - **brand/design convergence:** 55-65% ready
  - **external productization for other organisations:** 35-45% ready

These ranges are judgement calls based on the current files, deploys, and handoffs. They are not measured metrics.

## What Is Already Good Enough

- We no longer need to invent the architecture.
- We no longer need to prove that wiki -> packet -> hub can work.
- We do not need more generic planning before design starts.
- We have enough canon and enough integration to design the real public system now.

## What Still Blocks the “Designed Ecosystem” Feel

### Verified blockers

- Final hub/spoke/public-copy ownership decisions are still open in the canon registry.
- ACT hub and Empathy Ledger are content-aligned more than design-aligned.
- The knowledge system still lacks a few first-class page types that make operation obvious.
- External productization is still too ACT-specific in packet ownership, environment setup, and governance shape.

### Practical meaning

Right now the system reads as:

- **structurally coherent**
- **operationally promising**
- **visually related**
- not yet **ecosystem-native**

## The Exact Pages To Build Next

These are the pages that move the knowledge system from “working” to “a place we are happy with.”

1. **Living Ecosystem Map**
   - one page that clearly names primary hub, spokes, archive, sandbox, and ownership rules
2. **Source Packet Page**
   - a first-class human-readable page type for one canonical packet
3. **Source Bridge Page**
   - raw input -> bridge note -> canonical object path
4. **Canonical Project Page Template**
   - the stable wiki/public contract for a project
5. **Canonical Work Page Template**
   - the stable contract for portfolio objects across hub and spokes
6. **Review Queue Index**
   - one place showing editorial, cultural, consent, and release state
7. **Editorial / Cultural / Consent Review Surface**
   - one explicit approval page or page family
8. **Spoke Landing Template**
   - a shared pattern for JusticeHub / Harvest / Goods / BCV
9. **Works / Portfolio Index**
   - a stronger public system page for art, installations, commissions, and field works
10. **Release Verification Page**
   - one human-facing page showing what was checked before and after release

## Recommended Shift In Work Mode

### Move now

- Shift the main thread into:
  - design system convergence
  - page-template design
  - public IA tightening
  - work/portfolio page architecture

### Keep parallelized

- Leave agents on:
  - backend typing debt
  - admin route cleanup
  - environment hygiene
  - tenant/generalization prep

## What “Done Enough To Design” Means

The system is design-ready when these are true:

- one ecosystem map page exists
- one source packet page exists
- one source bridge page exists
- one canonical work template exists
- one review queue exists
- hub and EL share one visible public design grammar

That is the threshold where the knowledge system becomes a satisfying substrate for real brand/feel work rather than another backend project.

## Recommendation

Start design integration now.

Do **not** wait for total backend cleanliness.

Use the next design pass to lock:

- hub shell
- spoke shell
- packet/bridge/work page templates
- review/release visibility

That is the shortest path to a living ecosystem site that feels intentional rather than merely connected.
